# FEATURE-001: Авторизация по телефону и OTP-коду

## Цель

Клиент входит в приложение по номеру телефона: запрашивает одноразовый код
(OTP), вводит его, получает сессию (токен) и данные своего профиля. Без этого
экрана недоступны остальные функции (запись на слоты, брони, профиль).

## Требования

- `01-analysis/5-mobile-app-spec/SCR-001-registration.md` — экран регистрации/входа.
- `01-analysis/5-mobile-app-spec/09_Логики/LOGIC-001_OTP-авторизация.md` —
  логика запроса/проверки кода и хранения сессии.
- `01-analysis/api/auth/api.yaml`, `01-analysis/api/auth/models.yaml` — контракт API.

## Реализация

- `auth/AuthContracts.kt` — доменные контракты: `AuthRepository`
  (`requestCode`, `verifyCode`, `logout`), `SessionRepository` (хранение
  токена), `AuthFailure` (типизированные ошибки: неверный телефон/код, too
  many requests).
- `auth/data/KtorAuthRepository.kt` + `AuthDto.kt` + `AuthMappers.kt` —
  реализация поверх Ktor-клиента, маппинг DTO → доменные модели.
- `auth/data/DefaultSessionRepository.kt` — хранение токена через
  платформенный `SessionStorage` (expect/actual для Android/iOS/Web).
- `auth/presentation/AuthStore.kt` + `AuthScreen.kt` — MVI-стор и Compose-экран:
  ввод телефона → запрос кода → ввод OTP → сохранение токена → переход в
  каталог слотов.

## Промпты

См. `01-analysis/prompts/good-prompts.md` / `bad-prompts.md` — промпты для
генерации LOGIC-001 и контракта `auth` API. Плюс промпты, использованные
непосредственно в чате при реализации `AuthStore`/`AuthScreen`
*(приложить экспорт соответствующей переписки, если сохранена отдельно)*.

## Проверка

Функциональность реализована согласно контракту API и ТЗ (`SCR-001`,
`LOGIC-001`); отдельного цикла ручного регрессионного тестирования по всем
сценариям (запрос кода / неверный код / сохранение сессии между запусками) в
рамках этой сдачи проведено не было — отмечено как зона для последующей
проверки.
