Пример 1: Контракты OTP-авторизации

Промпт:


Спроектируй доменные контракты для авторизации по телефону с одноразовым
кодом (OTP) на Kotlin Multiplatform. Нужен интерфейс репозитория с
методами запроса кода, проверки кода и логаута; отдельный интерфейс для
хранения сессии (токена); и sealed-иерархия типизированных ошибок —
неверный телефон, неверный код, слишком много попыток, внешняя ошибка.
Контекст: клиентское MVP-приложение записи на слоты, авторизация — только
по телефону, без пароля.



Почему работает:


Задан домен (OTP-авторизация) и платформа (KMP) — ИИ сразу выбирает
идиоматичные конструкции (expect/actual, sealed interface).
Явно разделены два разных интерфейса (репозиторий действий и хранилище
сессии) — это подсказывает архитектурную границу, а не просто "напиши
авторизацию".
Перечислены конкретные типы ошибок вместо общего "обработай ошибки" —
ИИ не додумывает своё дерево исключений.


Ответ ИИ (фрагмент):


Предложил AuthRepository с тремя suspend-методами, SessionRepository
отдельным интерфейсом (без деталей хранения — под конкретную платформу), и
sealed interface AuthFailure с четырьмя вариантами, включая External
для проброса непредвиденных ошибок сети.




Пример 2: Фильтры каталога слотов

Промпт:


Спроектируй модель фильтров для списка слотов бронирования: диапазон дат,
набор типов маршрута, набор инструкторов, флаг "только доступные".
Фильтры внутри одной группы (например, несколько инструкторов) должны
комбинироваться через OR, а между разными группами — через AND. Добавь
универсальную пагинацию (limit/offset), которую можно переиспользовать и
для других списков (не только слотов).



Почему работает:


Логика комбинирования фильтров (OR внутри группы / AND между группами)
описана явно и однозначно — без этого ИИ мог бы выбрать любую
интерпретацию "фильтрации по нескольким критериям".
Отдельно попросили сделать пагинацию переиспользуемой — так вместо
специфичного SlotPageRequest получился обобщённый PageRequest/Page<T>.


Ответ ИИ (фрагмент):


Выдал data class SlotFilters с полями под каждую группу условий и
отдельные обобщённые PageRequest/Page<T>, которые затем оказалось
удобно переиспользовать в BookingRepository.listBookings.




Пример 3: Правила доступности мест и отмены брони

Промпт:


Реализуй чистые функции без побочных эффектов: (1) доступность мест как
min(free_seats, route.capacity_cap, 3); (2) отдельную проверку, что
число прокатных досок не превышает free_rental_boards (прокатный фонд
считается независимо от мест группы); (3) расчёт цены брони как
price * seats_count + rental_price * rental_count; (4) правило отмены —
>= 2 часов до старта слота — ранняя отмена (место освобождается),
< 2 часов — поздняя (место не освобождается), после старта отмена
недоступна. Покрой юнит-тестами граничный случай "ровно 2 часа".



Почему работает:


Задача разбита на 4 отдельные чистые функции — так проще проверить и
протестировать каждую независимо, вместо одного "монолитного" метода
calculateEverything().
Явно указан граничный случай ("ровно 2 часа") — именно такие границы
чаще всего становятся источником багов, и явное упоминание в промпте
гарантирует тест на этот случай.
Подчёркнуто, что места и прокатный фонд считаются независимо — без
этого уточнения легко получить код, где одно случайно влияет на другое.


Ответ ИИ (фрагмент):


Сгенерировал CancellationPolicy с функцией, принимающей момент отмены и
момент старта, и DomainPolicyTest с явным тестом
exactlyTwoHoursBeforeStartIsEarlyCancellation, проверяющим именно
граничное значение.
